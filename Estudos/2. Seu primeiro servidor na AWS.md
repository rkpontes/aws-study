# Etapa 2 — Seu primeiro servidor na AWS

Nesta etapa, você criará uma máquina virtual Ubuntu no **Amazon EC2**, acessará por SSH e instalará o Docker.

O EC2 fornece servidores virtuais chamados **instâncias**. Você escolhe sistema operacional, CPU, memória, armazenamento, rede e regras de acesso. ([Documentação AWS][1])

## 1. Conceitos que você precisa entender

### Instância

É uma máquina virtual executada dentro da AWS.

Ela terá:

* CPU e memória;
* disco EBS;
* sistema operacional;
* IP privado;
* opcionalmente, IP público;
* Security Groups;
* uma identidade dentro da AWS.

### Tipo de instância

O tipo define a capacidade da máquina.

Exemplos:

| Tipo        | Uso                                              |
| ----------- | ------------------------------------------------ |
| `t3.micro`  | Laboratórios e aplicações pequenas               |
| `t4g.micro` | Econômica, mas usa arquitetura ARM               |
| `t3.small`  | Aplicações pequenas com um pouco mais de memória |
| `m7i.large` | Aplicações gerais com mais capacidade            |
| `c7i.large` | Processamento intensivo                          |

Para este laboratório, use uma opção pequena indicada como elegível para os créditos ou benefícios disponíveis na sua conta.

Prefira **x86/AMD64** neste primeiro momento, porque reduz possíveis problemas de compatibilidade com imagens Docker.

### Security Group

É o firewall da instância.

Ele define:

* quais portas podem receber conexões;
* de quais endereços;
* quais conexões de saída são permitidas.

Security Groups controlam tráfego de entrada e saída da interface de rede da instância. ([Documentação AWS][2])

Para este laboratório, permita somente:

| Tipo | Porta | Origem               |
| ---- | ----: | -------------------- |
| SSH  |  `22` | My IP / seu IP atual |

Não use `0.0.0.0/0` na porta SSH. Isso deixaria a tentativa de conexão disponível para toda a internet.

### Key Pair

É o par de chaves utilizado para autenticar seu acesso SSH.

A AWS mantém a chave pública e você baixa a chave privada, normalmente como arquivo `.pem`. ([Documentação AWS][3])

Cuidados:

* não envie a chave para ninguém;
* não coloque o `.pem` no Git;
* não salve dentro do repositório do projeto;
* mantenha uma cópia segura;
* a AWS não permite baixar novamente a mesma chave privada.

### Elastic IP

É um endereço IPv4 público estático.

O IP público normal de uma EC2 pode mudar quando a instância é parada e iniciada. O Elastic IP permanece o mesmo enquanto estiver alocado à conta. ([Documentação AWS][4])

Para este laboratório, você pode estudá-lo, mas **não precisa utilizá-lo ainda**.

A AWS cobra atualmente pelos endereços IPv4 públicos, inclusive Elastic IP associado ou ocioso. A tarifa publicada é de **US$ 0,005 por hora por IPv4 público**, sujeita à região, créditos e condições da conta. ([Amazon Web Services, Inc.][5])

# 2. Criar a instância

Acesse:

```text
AWS Console → EC2 → Instances → Launch instances
```

## Configuração sugerida

### Name

```text
aws-study-ubuntu
```

### Application and OS Images

Escolha:

```text
Ubuntu Server 24.04 LTS
```

Use uma AMI oficial publicada pela Canonical.

### Instance type

Escolha uma instância pequena indicada na tela como elegível para os benefícios ou créditos da sua conta.

Sugestão:

```text
t3.micro
```

Use `t2.micro` apenas caso ela seja a opção oferecida ou elegível na região escolhida.

### Key Pair

Clique em:

```text
Create new key pair
```

Configure:

```text
Name: aws-study-key
Key pair type: ED25519
Private key format: .pem
```

Salve o arquivo, por exemplo, em:

```text
~/.ssh/aws-study-key.pem
```

No macOS:

```bash
mkdir -p ~/.ssh
mv ~/Downloads/aws-study-key.pem ~/.ssh/
chmod 400 ~/.ssh/aws-study-key.pem
```

O `chmod 400` impede que outros usuários do computador leiam ou modifiquem a chave.

### Network Settings

Mantenha inicialmente:

```text
VPC: default
Subnet: No preference
Auto-assign public IP: Enable
```

Crie um Security Group:

```text
Name: aws-study-ssh-sg
Description: SSH access from my IP
```

Regra de entrada:

```text
Type: SSH
Port: 22
Source: My IP
```

A documentação da AWS recomenda que o Security Group permita SSH a partir do seu endereço IP para conectar com um cliente SSH. ([Documentação AWS][6])

### Storage

Para o laboratório:

```text
8 GiB gp3
```

Não adicione discos extras.

Depois clique em:

```text
Launch instance
```

# 3. Esperar a instância ficar pronta

Acesse:

```text
EC2 → Instances
```

Espere aparecer:

```text
Instance state: Running
Status check: 2/2 checks passed
```

Selecione a instância e copie:

```text
Public IPv4 address
```

Exemplo fictício:

```text
18.231.100.50
```

# 4. Conectar por SSH

Para Ubuntu, o usuário padrão normalmente é:

```text
ubuntu
```

No terminal do Mac:

```bash
ssh -i ~/.ssh/aws-study-key.pem ubuntu@IP_PUBLICO
```

Exemplo:

```bash
ssh -i ~/.ssh/aws-study-key.pem ubuntu@18.231.100.50
```

Na primeira conexão, aparecerá uma confirmação parecida com:

```text
Are you sure you want to continue connecting?
```

Digite:

```text
yes
```

A AWS também apresenta o comando apropriado em:

```text
EC2 → Instances → sua instância → Connect → SSH client
```

Esse é o fluxo oficial recomendado para obter o comando de conexão correto. ([Documentação AWS][7])

## Configuração opcional no SSH

Para evitar escrever o comando completo, edite:

```bash
nano ~/.ssh/config
```

Adicione:

```sshconfig
Host aws-study
    HostName IP_PUBLICO
    User ubuntu
    IdentityFile ~/.ssh/aws-study-key.pem
```

Depois conecte usando:

```bash
ssh aws-study
```

Não versionar esse arquivo nem a chave privada.

# 5. Atualizar o Ubuntu

Dentro da EC2:

```bash
sudo apt update
sudo apt upgrade -y
```

Verifique o sistema:

```bash
cat /etc/os-release
```

Verifique recursos:

```bash
free -h
df -h
nproc
```

Você deverá observar:

* versão do Ubuntu;
* memória disponível;
* espaço em disco;
* quantidade de CPUs virtuais.

# 6. Instalar o Docker

Para aprendizagem, instale a versão oficial do Docker Engine por meio do repositório APT do Docker. Esse é o método recomendado na documentação oficial. ([Docker Documentation][8])

## Remover pacotes conflitantes

```bash
for pkg in docker.io docker-doc docker-compose podman-docker containerd runc; do
  sudo apt-get remove -y "$pkg"
done
```

Não há problema caso algum pacote não esteja instalado.

## Preparar o repositório

```bash
sudo apt-get update
sudo apt-get install -y ca-certificates curl
```

```bash
sudo install -m 0755 -d /etc/apt/keyrings
```

```bash
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
```

```bash
sudo chmod a+r /etc/apt/keyrings/docker.asc
```

Adicionar o repositório:

```bash
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}") stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

## Instalar o Docker

```bash
sudo apt-get update
```

```bash
sudo apt-get install -y \
  docker-ce \
  docker-ce-cli \
  containerd.io \
  docker-buildx-plugin \
  docker-compose-plugin
```

## Verificar o serviço

```bash
sudo systemctl status docker --no-pager
```

Também confira as versões:

```bash
sudo docker version
sudo docker compose version
```

# 7. Executar Docker sem `sudo`

Adicione o usuário `ubuntu` ao grupo Docker:

```bash
sudo usermod -aG docker ubuntu
```

Saia da sessão:

```bash
exit
```

Conecte novamente:

```bash
ssh aws-study
```

Ou:

```bash
ssh -i ~/.ssh/aws-study-key.pem ubuntu@IP_PUBLICO
```

Verifique:

```bash
docker version
```

Participar do grupo `docker` concede privilégios elevados sobre o servidor. Portanto, apenas usuários administrativos devem fazer parte desse grupo. As etapas pós-instalação são documentadas oficialmente pelo Docker. ([Docker Documentation][9])

# 8. Teste final

Execute:

```bash
docker run --rm hello-world
```

O fluxo será:

1. Docker procura a imagem localmente;
2. não encontra;
3. baixa a imagem;
4. cria o container;
5. executa o programa;
6. remove o container por causa de `--rm`.

A saída deve conter:

```text
Hello from Docker!
```

Verifique os containers:

```bash
docker ps
```

Como o container já terminou e foi removido, a lista provavelmente estará vazia.

# 9. Exercício adicional

Execute um servidor Nginx:

```bash
docker run -d \
  --name nginx-study \
  --restart unless-stopped \
  -p 80:80 \
  nginx:alpine
```

Para acessá-lo externamente, adicione temporariamente ao Security Group:

| Tipo | Porta | Origem      |
| ---- | ----: | ----------- |
| HTTP |  `80` | `0.0.0.0/0` |

```
Na Console da AWS:

Acesse EC2 → Instances.
Selecione a instância aws-study-ubuntu.
Abra a aba Security.
Em Security groups, clique no nome do grupo associado, por exemplo aws-study-ssh-sg.
Abra a aba Inbound rules.
Clique em Edit inbound rules.
Clique em Add rule.
Configure:
Campo	Valor
Type	HTTP
Protocol	TCP
Port range	80
Source type	Anywhere-IPv4
Source	0.0.0.0/0
Description	Temporary nginx test
Clique em Save rules.

As alterações do Security Group são aplicadas automaticamente à instância; não é necessário reiniciá-la.
```

Depois abra:

```text
http://IP_PUBLICO
```

Você deverá visualizar a página inicial do Nginx.

Confira:

```bash
docker ps
docker logs nginx-study
```

Remova o laboratório:

```bash
docker rm -f nginx-study
```

Depois exclua a regra HTTP do Security Group.

# 10. Erros comuns

## `Permission denied (publickey)`

Verifique:

```bash
chmod 400 ~/.ssh/aws-study-key.pem
```

Confirme que está usando:

```text
ubuntu
```

e não:

```text
root
```

Também confirme que a chave corresponde ao Key Pair usado na criação da instância.

## `Connection timed out`

Possíveis causas:

* porta 22 não liberada;
* regra do Security Group aponta para um IP antigo;
* instância ainda inicializando;
* instância sem IPv4 público;
* IP público copiado incorretamente.

Se sua internet mudou de IP, atualize a origem da regra SSH para:

```text
My IP
```

A documentação de troubleshooting da AWS aponta o Security Group como uma das primeiras verificações em casos de timeout. ([Documentação AWS][10])

## `docker: permission denied`

Confirme:

```bash
groups
```

A saída precisa conter:

```text
docker
```

Caso não contenha:

```bash
sudo usermod -aG docker ubuntu
exit
```

Depois reconecte.

# 11. Controle de custos

Uma EC2 pode gerar custo enquanto estiver em execução, mesmo sem receber acessos. O disco EBS também pode permanecer cobrando depois que a instância é parada.

Para interromper temporariamente:

```text
EC2 → Instances → Instance state → Stop instance
```

Para excluir definitivamente:

```text
EC2 → Instances → Instance state → Terminate instance
```

Diferença:

* **Stop:** desliga a máquina, mas preserva o disco;
* **Start:** liga novamente, possivelmente com outro IP público;
* **Reboot:** reinicia sem trocar a infraestrutura;
* **Terminate:** exclui a instância definitivamente.

Contas criadas a partir de **15 de julho de 2025** usam o modelo mais recente de Free Tier, baseado em plano gratuito e créditos, normalmente válido por até seis meses ou até os créditos acabarem. Verifique diretamente o painel de Billing da sua conta. ([Documentação AWS][11])
