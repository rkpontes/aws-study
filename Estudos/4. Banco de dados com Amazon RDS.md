# Etapa 4 — Banco de dados com Amazon RDS

Nesta etapa, vamos remover o PostgreSQL da EC2 e utilizar o **Amazon RDS for PostgreSQL**.

A arquitetura ficará assim:

```text
Internet
   ↓
EC2
   ↓ porta 5432 privada
RDS PostgreSQL
```

O RDS deve ficar **privado**, acessível somente pela EC2. A AWS recomenda esse modelo para aplicações executadas dentro da mesma VPC. ([Documentação AWS][1])

---

## 1. Entendendo o problema atual

Hoje você provavelmente tem algo semelhante a:

```text
EC2
├── Backend
├── Frontend
└── PostgreSQL
```

Isso é funcional para estudos e aplicações pequenas, mas cria alguns problemas:

* aplicação e banco disputam memória, CPU e armazenamento;
* uma falha na EC2 pode afetar aplicação e banco;
* substituir ou excluir a EC2 pode colocar os dados em risco;
* backups e recuperação ficam sob sua responsabilidade;
* escalar aplicação e banco separadamente fica mais difícil.

Com o RDS:

```text
EC2
├── Backend
└── Frontend

RDS
└── PostgreSQL
```

O RDS gerencia tarefas como backups, correções do banco, detecção de falhas e recuperação. ([Documentação AWS][2])

# Parte 1 — Preparar os Security Groups

O melhor modelo é ter dois Security Groups:

```text
sg-ec2-app
sg-rds-postgres
```

O Security Group do banco não deve receber `0.0.0.0/0`.

## Security Group da EC2

Exemplo:

| Tipo  | Porta | Origem      |
| ----- | ----: | ----------- |
| SSH   |    22 | Seu IP      |
| HTTP  |    80 | `0.0.0.0/0` |
| HTTPS |   443 | `0.0.0.0/0` |

A saída pode continuar com a configuração padrão:

```text
All traffic → 0.0.0.0/0
```

## Security Group do RDS

Crie um Security Group chamado:

```text
sg-rds-postgres
```

Adicione esta regra de entrada:

| Tipo       | Porta | Origem       |
| ---------- | ----: | ------------ |
| PostgreSQL |  5432 | `sg-ec2-app` |

Observe que a origem não é um IP. É o próprio Security Group da EC2:

```text
PostgreSQL
TCP 5432
Source: sg-ec2-app
```

Isso significa:

> Qualquer recurso associado ao `sg-ec2-app` pode tentar se conectar ao PostgreSQL.

A AWS utiliza exatamente esse modelo ao conectar automaticamente uma EC2 ao RDS: o Security Group do RDS recebe uma regra de entrada cuja origem é o Security Group associado à EC2. ([Documentação AWS][3])

# Parte 2 — Criar o PostgreSQL no RDS

Entre no console da AWS:

```text
RDS → Databases → Create database
```

## Método de criação

Selecione:

```text
Standard create
```

## Engine

Selecione:

```text
PostgreSQL
```

Escolha uma versão compatível com seu PostgreSQL atual.

Por exemplo, caso esteja usando:

```yaml
image: postgres:16-alpine
```

Use PostgreSQL 16 no RDS.

Isso reduz o risco de incompatibilidades durante a restauração.

## Template

Para estudo:

```text
Free tier
```

Caso essa opção não esteja disponível para sua conta ou configuração:

```text
Dev/Test
```

Confira o resumo de custos antes de criar o recurso.

## Identificação

Exemplo:

```text
DB instance identifier: app-postgres
```

## Credenciais

Exemplo:

```text
Master username: postgres
Master password: uma-senha-forte
```

Não coloque essa senha diretamente no Git ou dentro do `docker-compose.yml`.

## Classe da instância

Para estudo, escolha a menor classe elegível apresentada pelo console, normalmente alguma opção burstable, como:

```text
db.t4g.micro
```

A disponibilidade e a elegibilidade para benefícios gratuitos podem variar por região e conta, portanto confirme o custo mostrado no console.

## Armazenamento

Para estudo:

```text
Storage type: General Purpose SSD
Allocated storage: 20 GiB
```

Você pode manter o autoscaling desativado inicialmente para ter mais previsibilidade de custo.

# Parte 3 — Configurar conectividade

Esta é a parte mais importante.

## VPC

Selecione a mesma VPC utilizada pela EC2:

```text
VPC: mesma VPC da EC2
```

Por exemplo:

```text
vpc-0123456789
```

## Public access

Selecione:

```text
Public access: No
```

O banco não precisa estar exposto na internet, pois a aplicação se conecta pela rede privada da AWS. Uma instância privada normalmente pode ser acessada por recursos, como EC2, que estejam dentro da mesma VPC. ([Documentação AWS][1])

## VPC Security Group

Selecione:

```text
Choose existing
```

Depois:

```text
sg-rds-postgres
```

## Availability Zone

Pode deixar:

```text
No preference
```

## Porta

Mantenha:

```text
5432
```

# Parte 4 — Configurar backups

Na seção de configurações adicionais, habilite:

```text
Automated backups: Enabled
```

Para estudo, uma configuração razoável é:

```text
Backup retention period: 7 days
```

O RDS faz backups automáticos durante a janela configurada e permite recuperação para um ponto no tempo dentro do período de retenção. O backup automático abrange a instância, e não apenas um banco individual. ([Documentação AWS][4])

Você também pode configurar:

```text
Backup window: No preference
```

Para produção, selecione uma janela de menor utilização.

## Proteção contra exclusão

Durante o aprendizado, você pode deixar desabilitada para facilitar a remoção:

```text
Deletion protection: Disabled
```

Em produção, mantenha:

```text
Deletion protection: Enabled
```

Revise tudo e clique em:

```text
Create database
```

Aguarde o status mudar para:

```text
Available
```

# Parte 5 — Obter o endpoint do RDS

Abra:

```text
RDS → Databases → app-postgres
```

Na seção **Connectivity & security**, procure:

```text
Endpoint
Port
```

Exemplo:

```text
app-postgres.abc123.us-east-1.rds.amazonaws.com
5432
```

O endpoint é um endereço DNS. Não configure sua aplicação com um IP fixo do RDS.

A AWS disponibiliza endpoint, porta e demais dados de conexão na página da instância. ([Documentação AWS][5])

# Parte 6 — Testar a conexão pela EC2

Entre na EC2:

```bash
ssh -i sua-chave.pem ubuntu@IP_PUBLICO_DA_EC2
```

Instale o cliente PostgreSQL:

```bash
sudo apt update
sudo apt install postgresql-client -y
```

Teste a porta:

```bash
nc -zv \
  app-postgres.abc123.us-east-1.rds.amazonaws.com \
  5432
```

Resultado esperado:

```text
Connection to ... 5432 port [tcp/postgresql] succeeded!
```

Depois teste com o `psql`:

```bash
psql \
  --host=app-postgres.abc123.us-east-1.rds.amazonaws.com \
  --port=5432 \
  --username=postgres \
  --dbname=postgres
```

Digite a senha quando solicitado.

Dentro do PostgreSQL:

```sql
SELECT version();
```

Para sair:

```text
\q
```

## Caso dê timeout

Verifique:

1. EC2 e RDS estão na mesma VPC?
2. O RDS está com o `sg-rds-postgres`?
3. O Security Group do RDS permite a porta `5432`?
4. A origem da regra é o Security Group da EC2?
5. O status do RDS está como `Available`?

# Parte 7 — Criar o banco da aplicação

Conecte ao RDS:

```bash
psql \
  --host=ENDPOINT_DO_RDS \
  --username=postgres \
  --dbname=postgres
```

Crie o banco:

```sql
CREATE DATABASE etapa_3_app;
```

Para listar:

```text
\l
```

Para aplicações reais, é melhor criar um usuário específico, em vez de usar o usuário administrador do RDS:

```sql
CREATE USER etapa_3_app_user
WITH PASSWORD 'senha-forte-da-aplicacao';
```

Conceda acesso:

```sql
GRANT ALL PRIVILEGES
ON DATABASE etapa_3_app
TO etapa_3_app_user;
```

Conecte ao banco:

```text
\c etapa_3_app
```

Conceda acesso ao schema:

```sql
GRANT ALL ON SCHEMA public
TO etapa_3_app_user;
```

# Parte 8 — Carregar a estrutura inicial da etapa 3

Como a `etapa-3-app` não usa banco de dados, aqui não existe migração de um PostgreSQL antigo.

Em vez disso, vamos importar um dump SQL inicial com a estrutura mínima da aplicação.

Arquivo criado no projeto:

```text
Projetos/etapa-4/dump-etapa-4.sql
```

Esse arquivo cria a tabela `app_messages` e insere uma mensagem inicial que pode ser usada pela próxima versão do backend.

Antes de importar, copie o arquivo para a EC2 ou execute o comando a partir de uma máquina que tenha acesso ao RDS pela rede correta.

Exemplo usando `scp`:

```bash
scp -i sua-chave.pem \
  Projetos/etapa-4/dump-etapa-4.sql \
  ubuntu@IP_PUBLICO_DA_EC2:/home/ubuntu/
```

Depois, na EC2, importe o arquivo:

```bash
psql \
  --host=ENDPOINT_DO_RDS \
  --port=5432 \
  --username=postgres \
  --dbname=etapa_3_app \
  --file=/home/ubuntu/dump-etapa-4.sql
```

Se preferir usar o usuário da aplicação:

```bash
psql \
  --host=ENDPOINT_DO_RDS \
  --port=5432 \
  --username=etapa_3_app_user \
  --dbname=etapa_3_app \
  --file=/home/ubuntu/dump-etapa-4.sql
```

Como este arquivo está em SQL plano, ele pode ser lido, revisado e reaplicado com facilidade durante os estudos.

# Parte 10 — Atualizar a aplicação

Na `etapa-3-app`, o backend ainda não usa banco. A próxima evolução é adicionar as variáveis de conexão para que a aplicação leia e grave dados no RDS.

Entre na EC2 por SSH:

```bash
ssh -i sua-chave.pem ubuntu@IP_PUBLICO_DA_EC2
```

Entre na pasta do projeto:

```bash
cd /caminho/do/projeto/etapa-3-app
```

Entre na pasta do backend:

```bash
cd backend
```

Crie o arquivo `.env`:

```bash
nano .env
```

Conteúdo sugerido:

```env
DB_HOST=app-postgres.abc123.us-east-1.rds.amazonaws.com
DB_PORT=5432
DB_NAME=etapa_3_app
DB_USER=etapa_3_app_user
DB_PASSWORD=senha-forte-da-aplicacao
```

Salve no `nano` com:

```text
Ctrl + O
Enter
Ctrl + X
```

Confirme:

```bash
cat .env
```

Se você preferir uma variável única, monte a URL com os mesmos nomes usados acima:

```env
DATABASE_URL=postgresql://etapa_3_app_user:SENHA@app-postgres.abc123.us-east-1.rds.amazonaws.com:5432/etapa_3_app
```

Se a senha possuir caracteres especiais, como `@`, `#`, `/` ou `:`, eles precisam ser codificados quando usados em uma URL.

# Parte 11 — Alterar o backend para usar o RDS

No backend atual, a rota `/api/message` devolve uma mensagem fixa. Agora ela deve ler a mensagem inicial gravada na tabela `app_messages`.

Instale as dependências:

```bash
cd /caminho/do/projeto/etapa-3-app/backend
sudo apt update
sudo apt install -y nodejs npm
npm install pg dotenv
```

Exemplo de implementação para o `server.js`:

```js
require("dotenv").config();
const express = require("express");
const { Pool } = require("pg");

const app = express();
const port = 3000;

const pool = new Pool({
    host: process.env.DB_HOST,
    port: Number(process.env.DB_PORT || 5432),
    database: process.env.DB_NAME,
    user: process.env.DB_USER,
    password: process.env.DB_PASSWORD
});

app.get("/health", async (request, response) => {
    try {
        await pool.query("SELECT 1");

        response.json({
            status: "ok",
            service: "backend",
            database: "connected"
        });
    } catch (error) {
        response.status(500).json({
            status: "error",
            service: "backend",
            database: "disconnected"
        });
    }
});

app.get("/api/message", async (request, response) => {
    try {
        const result = await pool.query(`
            SELECT title, content, created_at
            FROM app_messages
            WHERE slug = 'mensagem-principal'
            LIMIT 1
        `);

        if (result.rowCount === 0) {
            return response.status(404).json({
                error: "Mensagem não encontrada"
            });
        }

        const message = result.rows[0];

        response.json({
            message: message.content,
            title: message.title,
            createdAt: message.created_at
        });
    } catch (error) {
        response.status(500).json({
            error: "Erro ao consultar o banco de dados"
        });
    }
});

app.listen(port, "0.0.0.0", () => {
    console.log(`Backend executando na porta ${port}`);
});
```

Se quiser manter a URL única em vez de variáveis separadas, o `Pool` pode ser criado assim:

```js
const pool = new Pool({
    connectionString: process.env.DATABASE_URL
});
```

Como o arquivo `.env` está em `backend/.env` e o `server.js` também está em `backend`, o `require("dotenv").config()` funciona sem ajustes extras.

# Parte 13 — Validar a leitura do banco

Teste o backend:

```bash
docker exec -it etapa-3-backend wget -qO- http://127.0.0.1:3000/health
curl http://localhost/api/message
```

Resultado esperado para o healthcheck interno do backend:

```json
{
  "status": "ok",
  "service": "backend",
  "database": "connected"
}
```

Resultado esperado para `/api/message`:

```json
{
  "message": "Backend conectado ao PostgreSQL no Amazon RDS.",
  "title": "Mensagem inicial",
  "createdAt": "2026-07-17T00:00:00.000Z"
}
```

Observe a diferença:

```text
/health
```

é uma rota do backend acessível dentro do container, enquanto:

```text
/api/message
```

passa pelo Nginx e fica disponível pelo IP público da EC2.

Com a configuração atual, esta URL não funciona:

```text
http://IP_PUBLICO_DA_EC2/api/health
```

porque o backend expõe:

```text
/health
```

e não:

```text
/api/health
```

O `docker compose` continua com apenas dois serviços:

```text
backend
frontend
```

# Parte 14 — Criar um snapshot manual

Após confirmar a migração:

```text
RDS
→ Databases
→ app-postgres
→ Actions
→ Take snapshot
```

Nome sugerido:

```text
etapa-4-app-carga-inicial-2026-07-17
```

Existem duas proteções diferentes:

```text
Backup automático
Snapshot manual
```

O backup automático segue o período de retenção configurado. Já o snapshot manual permanece até ser excluído manualmente. O RDS oferece as duas modalidades. ([Documentação AWS][7])

# Fluxo seguro da migração

Para evitar perda de dados, faça nesta ordem:

```text
1. Criar RDS
2. Configurar Security Groups
3. Testar conexão EC2 → RDS
4. Criar banco e usuário
5. Importar `Projetos/etapa-4/dump-etapa-4.sql`
6. Validar as tabelas e os dados iniciais
7. Criar o `.env` na EC2
8. Atualizar o backend para usar PostgreSQL
9. Reiniciar aplicação
10. Validar leitura e escrita
11. Criar snapshot
```
