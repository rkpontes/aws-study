## Etapa 1 — Conhecendo e protegendo a conta AWS

**Duração sugerida:** 2 a 3 dias
**Carga estimada:** 4 a 6 horas
**Projeto prático:** preparar uma conta AWS segura para os próximos laboratórios.

### 1. Conceitos fundamentais

#### Regions

Uma **Region** é uma área geográfica independente onde a AWS mantém sua infraestrutura, como:

* `sa-east-1`: São Paulo
* `us-east-1`: Norte da Virgínia
* `us-east-2`: Ohio
* `eu-west-1`: Irlanda

A escolha da Region influencia:

* Latência para os usuários.
* Preço dos serviços.
* Serviços disponíveis.
* Requisitos legais e residência dos dados.
* Estratégia de recuperação de desastres.

Para os estudos, utilize preferencialmente **`sa-east-1`**, por estar mais próxima do Brasil. Contudo, alguns laboratórios podem usar `us-east-1` por disponibilidade ou menor custo.

> Recursos criados em uma Region normalmente não aparecem automaticamente em outra.

#### Availability Zones

Uma **Availability Zone — AZ** é uma localização isolada dentro de uma Region. Uma AZ pode conter um ou mais data centers com energia, rede e conectividade redundantes. Cada Region possui múltiplas AZs. ([Documentação AWS][1])

Exemplo:

```text
Region: sa-east-1
├── sa-east-1a
├── sa-east-1b
└── sa-east-1c
```

Em uma arquitetura de produção, aplicações críticas costumam ser distribuídas entre pelo menos duas AZs:

```text
Usuário
   │
Load Balancer
   ├── Aplicação na AZ A
   └── Aplicação na AZ B
```

Isso permite que a aplicação continue disponível caso uma AZ apresente falha.

#### IAM

O **AWS Identity and Access Management — IAM** controla:

* Quem pode acessar a conta.
* Como essa pessoa ou sistema se autentica.
* Quais recursos podem ser acessados.
* Quais ações podem ser executadas.

Conceitos fundamentais:

| Conceito | Função                                             |
| -------- | -------------------------------------------------- |
| User     | Identidade permanente de uma pessoa ou integração  |
| Group    | Agrupamento de usuários com permissões semelhantes |
| Role     | Identidade assumida temporariamente                |
| Policy   | Documento JSON que define permissões               |
| MFA      | Segundo fator de autenticação                      |

Exemplo simplificado de uma policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:ListAllMyBuckets",
      "Resource": "*"
    }
  ]
}
```

A regra central é o **princípio do menor privilégio**: uma identidade deve possuir apenas as permissões necessárias para executar seu trabalho.

Para ambientes profissionais, a recomendação atual da AWS é preferir **IAM Identity Center, federação e credenciais temporárias** para usuários humanos. O usuário IAM continua válido para aprender os fundamentos e para alguns casos específicos, mas não deve ser tratado como a única solução de identidade para uma organização. ([Documentação AWS][2])

#### Conta root

O usuário root é criado junto com a conta AWS e possui acesso irrestrito.

Ele deve ser utilizado somente para operações que realmente exigem root, como determinadas configurações da conta. A AWS recomenda:

* Ativar MFA.
* Não criar access keys para o root.
* Não utilizar root no trabalho diário.
* Proteger os mecanismos de recuperação da conta. ([Documentação AWS][3])

#### AWS Management Console

A Console é a interface web da AWS. Durante esta etapa, localize:

* Seletor de Region.
* IAM.
* Billing and Cost Management.
* AWS Budgets.
* Free Tier.
* CloudShell.
* Resource Groups e Tag Editor.

Observe que a Console é apenas uma das formas de operar a AWS. Os mesmos recursos podem ser gerenciados por:

* AWS CLI.
* SDKs.
* APIs.
* CloudFormation.
* AWS CDK.
* Terraform.

#### Billing

A área de Billing permite acompanhar:

* Custos acumulados.
* Custos por serviço.
* Previsão mensal.
* Créditos promocionais.
* Uso do Free Tier.
* Faturas.
* Budgets e alertas.

O AWS Budgets permite criar limites e receber alertas quando o custo real ou previsto se aproxima do valor configurado. Os dados do Budget não são instantâneos e normalmente são atualizados algumas vezes ao dia. ([Amazon Web Services, Inc.][4])

#### Free Tier

Para contas novas, o modelo atual do AWS Free Tier pode incluir um plano gratuito baseado em créditos e tempo limitado. A AWS informa que o plano gratuito pode terminar quando os créditos forem consumidos ou quando o período disponível chegar ao fim. As condições exibidas durante a criação da conta devem ser consideradas a fonte definitiva para sua conta. ([Documentação AWS][5])

O termo “Free Tier” não significa que todos os recursos sejam ilimitadamente gratuitos. É necessário conferir:

* Limite de horas.
* Limite de armazenamento.
* Quantidade de requisições.
* Transferência de dados.
* Region utilizada.
* Prazo da oferta.

## 2. Laboratório prático

### Passo 1 — Criar a conta AWS

Durante o cadastro:

1. Utilize um e-mail ao qual você tenha acesso permanente.
2. Configure uma senha forte e exclusiva.
3. Escolha entre os planos apresentados pela AWS.
4. Registre os dados necessários para validação.
5. Armazene o ID da conta em local seguro.

Não compartilhe:

* Senha da conta.
* Código de MFA.
* Access Key ID.
* Secret Access Key.
* Session Token.

### Passo 2 — Proteger o root com MFA

Entre como root apenas para realizar a configuração inicial:

1. Abra as credenciais de segurança da conta.
2. Localize **Multi-factor authentication — MFA**.
3. Selecione **Assign MFA device**.
4. Cadastre um autenticador ou chave de segurança.
5. Valide os códigos solicitados.
6. Saia da conta root.

A AWS permite associar múltiplos dispositivos MFA. Ter uma alternativa segura reduz o risco de bloqueio caso o dispositivo principal seja perdido. ([Documentação AWS][6])

### Passo 3 — Configurar alertas de custo

Na área de Billing:

1. Ative os alertas de uso do Free Tier.
2. Abra **AWS Budgets**.
3. Crie um orçamento mensal.
4. Cadastre seu e-mail para notificações.
5. Configure alertas de custo real e previsto.

Sugestão inicial para uma conta de estudos:

```text
Budget mensal: US$ 5
Alerta real: 50%
Alerta real: 80%
Alerta previsto: 100%
```

Crie também um **Zero Spend Budget**, quando essa opção estiver disponível. Os alertas de Free Tier podem notificar quando determinado uso ultrapassa 85% do limite, mas não substituem o acompanhamento do Billing. ([Documentação AWS][7])

> Budget é um mecanismo de alerta. Ele não interrompe automaticamente todos os recursos quando o limite é alcançado.

### Passo 4 — Criar o usuário IAM de laboratório

Para cumprir o objetivo didático desta etapa:

1. Entre no serviço IAM.
2. Crie um usuário, por exemplo:

```text
raphael-admin-lab
```

3. Habilite acesso à AWS Management Console.
4. Associe temporariamente uma permissão administrativa.
5. Ative MFA nesse usuário.
6. Teste o login pelo endereço específico de acesso IAM.
7. Saia do root e continue utilizando o usuário IAM.

Para uma conta pessoal de laboratório, `AdministratorAccess` simplifica as primeiras etapas. Isso não representa a configuração recomendada para produção. Posteriormente, a permissão deverá ser substituída por roles e policies de menor privilégio.

### Passo 5 — Instalar a AWS CLI

No macOS com Homebrew:

```bash
brew install awscli
```

Valide:

```bash
aws --version
```

O resultado será semelhante a:

```text
aws-cli/2.x.x Python/3.x Darwin/...
```

### Passo 6 — Configurar credenciais

Para cumprir o laboratório básico com IAM User, crie uma access key especificamente para uso na CLI. Não utilize credenciais do root e não coloque as chaves no código-fonte. ([Documentação AWS][8])

Configure um profile nomeado:

```bash
aws configure --profile lab
```

Informe:

```text
AWS Access Key ID:     credencial fornecida pelo IAM
AWS Secret Access Key: credencial fornecida pelo IAM
Default region name:   sa-east-1
Default output format: json
```

As credenciais serão armazenadas normalmente em:

```text
~/.aws/credentials
```

As configurações ficam em:

```text
~/.aws/config
```

Nunca adicione esses arquivos a um repositório Git.

### Passo 7 — Testar a identidade

Execute:

```bash
aws sts get-caller-identity --profile lab
```

Resposta esperada:

```json
{
  "UserId": "...",
  "Account": "123456789012",
  "Arn": "arn:aws:iam::123456789012:user/raphael-admin-lab"
}
```

Esse comando confirma:

* Que a CLI está instalada.
* Que as credenciais são válidas.
* Qual conta está sendo utilizada.
* Qual identidade está executando os comandos.

Liste as Regions disponíveis:

```bash
aws ec2 describe-regions \
  --profile lab \
  --query "Regions[].RegionName" \
  --output table
```

Teste a Region configurada:

```bash
aws configure get region --profile lab
```

## 4. Exercícios de fixação

Responda com suas próprias palavras:

1. Qual é a diferença entre Region e Availability Zone?
2. Por que uma aplicação de produção deve utilizar mais de uma AZ?
3. Por que o usuário root não deve ser utilizado diariamente?
4. Qual é a diferença entre IAM User, Role e Policy?
5. O que significa menor privilégio?
6. Por que o Free Tier não elimina completamente o risco de cobrança?
7. Qual comando identifica a conta e o usuário utilizados pela CLI?
8. Onde a AWS CLI armazena as credenciais locais?
